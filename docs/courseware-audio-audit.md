# KidsLab 课件交互音效审计

审计日期：2026-07-22

## 结论

当前 `src/` 下共有 50 个课件目录。排除非游戏欢迎页 `welcome` 后，49 个游戏课件的静态扫描结果如下：

| 状态 | 数量 | 说明 |
| --- | ---: | --- |
| P0：高频音源风险 | 1 | 已有音效，但触发方式可能在拖动或物理循环中连续创建音源，需优先修复 |
| P1：已有音效但控制不完整 | 12 | 有核心音效，但没有用户可见的静音开关；其中 1 个还需优化连续输入音效 |
| P2：完全没有音频实现 | 22 | 未发现真实音频 API，需要按核心玩法补齐交互、结果与通关反馈 |
| 已有音效和静音控制，待回归 | 14 | 暂无明确源码缺口，但仍需按新规范人工听测 |
| 合计 | 49 | 所有游戏课件均已归类 |

这是一轮**源码静态审计**，不是听感验收。它不能证明实际响度合适、移动端一定出声、成功/失败语义清楚，也不能证明每条完整玩法路径都有音效。

## 审计口径

- 扫描 `src/<id>/` 下的 HTML、JavaScript 和 ES Module 文件，排除 `vendor/`。
- 只有出现 `AudioContext`、`webkitAudioContext`、`createOscillator()`、`createBufferSource()`、`new Audio()`、`<audio>` 或媒体 `.play()` 等真实发声实现，才计为“已有音频”。仅有 `sound`、`音效` 等文案不计入。
- 对 27 个已有音频的游戏继续检查静音状态、可见控件，以及拖动、动画帧和物理循环中的高频触发点。
- “有静音状态变量”不等于“有用户可见的静音开关”；例如 `shadow-theater` 和 `voxel-studio` 只有内部 `soundOn` 保护，没有可见控件。

## 完成标准

每个课件改进后都应满足 `AGENT.md` 的音效条款，并至少人工验证：

1. 按玩法实际存在的状态，核心直接操作、正确/成功、错误/无效操作、关卡或完整闭环通关有语义明确且彼此可辨的反馈；纯观察型玩法没有错误状态时不强造失败音。
2. 音效不是唯一反馈，关闭声音后仍可依靠动画、文字、图标或形状完成玩法。
3. 有可见、≥44×44px、支持中英双语 `aria-label` 和正确 `aria-pressed` 的静音开关；静音能关闭后续一次性声音和循环/环境音。
4. 首次用户手势前不播放；WebAudio 不可用或恢复失败时静默降级，玩法和控制台不受影响。
5. 快速拖动、连点、持续碰撞和长时间运行不会在 `pointermove`、动画帧或物理帧中堆叠音源。
6. 在桌面与手机上完整玩一遍并人工听测；现有 Playwright smoke 只检查加载和控制台错误，不验证实际出声与听感。

## P0：先消除高频音源风险

| 课件 | 已发现问题 | 最小改进 |
| --- | --- | --- |
| `magnet-hockey` 磁力冰球 | `pointer()` 在 `pointerdown` 和每次拖动 `pointermove` 中都创建短音；障碍碰撞音位于 60Hz 物理循环内，持续重叠时可能重复触发；页面没有静音开关 | 拖动改为只在拾起/释放或节流后的状态边界发声；碰撞只在“进入碰撞”时触发或增加冷却；增加可见、可访问且能关闭全部声音的静音开关 |
